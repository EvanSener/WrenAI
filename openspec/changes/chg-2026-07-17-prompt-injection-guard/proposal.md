# 为什么

Wren 的自然语言入口目前会把用户问题直接交给 Agent 提示模板或语义召回，
项目级查询执行也可以沿用全局默认的非严格 SQL 策略。攻击者因此可能诱导 Agent
泄露连接信息、内部提示词和实现细节，或者绕过 MDL 语义层直接访问物理表及执行
有副作用的 SQL。

# 变更内容

- 在 `wren_project.yml` 增加可选、默认关闭的 `security` 项目策略。
- 在 Ask、Graph 自然语言查询和 Cube 召回入口执行确定性的业务数据边界检查。
- 对启用策略的项目，在执行前强制只读 SQL、单语句、MDL 对象白名单和危险函数
  拒绝；显式连接参数也不能降低项目安全策略。
- 记录不含原始问题、SQL 和凭据的结构化 JSONL 审计事件。
- `maxcompute-local` 审计使用项目内 `.wren/audit/security.jsonl`，由 Wren 进程
  直接写入；落盘故障降级为告警，不让审计存储可用性阻断合法业务查询。
- 强化 Ask 模板的可信指令与不可信用户内容隔离，但不依赖模型完成授权判断。

# 不做什么

- 不宣称通过关键词检测即可解决所有 Prompt 注入。
- 不修改未启用 `security` 的既有项目行为。
- 不改 Semantic Graph、Cube、Memory、MaxCompute 分区和 MDL 编译规则。
- 不把数据库凭据写入项目文件、Prompt 或审计日志。
